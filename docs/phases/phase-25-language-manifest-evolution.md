# Phase 25 — Language & Manifest Evolution

## Status: Done

## Motivation

As MCL has been applied to real missions, several structural decisions have clarified:

- The `|>` operator carries F# baggage. `->` ("passes to") is more neutral and readable.
- OCI expert declarations (`from`, `version`) in `mission.mcl` conflate infrastructure with reasoning. They belong in a separate manifest.
- Provider config (apiKey, model, endpoint) changes together as a unit — named profiles in a manifest are the right model.
- Expert resolution needs a clear local-first precedence rule with a global shared cache.
- Parallel execution belongs as a first-class block construct, not an operator convention.

The result is a two-file model:

```
missions/build-operator/
  mission.mcl     ← thinking model + domain variables only
  forge.toml      ← experts manifest + provider profiles
  mcl.lock        ← resolved hashes (generated)
```

## What does not change

- `mcl.lock` format and discipline — generated by `forge init`, committed, never hand-edited
- `expert.md` file format — frontmatter + system prompt unchanged
- `with { }` clause on steps — still the per-step context override mechanism
- `loop N` — unchanged
- `runs/` output structure — unchanged
